## 🧠 وكيل النمو الذكي (Growth Agent) — مشروع استراتيجي كبير (17 يوليو 2026)
> التموضع الجديد: Mawq3i ما بتبيع "متجر"، بتبيع نمو مُدار بالذكاء الاصطناعي. المواصفة الكاملة (الرؤية، مكونات النظام، قاعدة معايير السوق 2026، المخاطر) موجودة بـ `docs/growth-agent-spec.md`. هون بس نتابع تنفيذ المراحل.
>
> ⚠️ **يستاهل تحقق:** صار عنا 5 Cron jobs بـ `vercel.json` (sync، ads-sync، diagnose، capture-results، monthly-plan). خطة Vercel "Hobby" فيها حدود على عدد/تكرار الـ Cron jobs بتختلف حسب الخطة — تأكد إنه حسابكم يدعمهم قبل ما تعتمد عليهم بالكامل (تحقق من Vercel dashboard → Settings → Cron Jobs بعد أول نشر).
>
> 📍 **توضيح صفحتين مختلفتين بقصد:** `/dashboard/growth` (لوحة التاجر — المنتج الفعلي، فيها التقرير الشهري + التشخيصات + أزرار الموافقة) مقابل `/admin/growth-agent` (لوحة الأدمن — أداة تحقق تقنية لعثمان بس، يشوف فيها بيانات خام لأي متجر ومعايير السوق). الاتنين موجودين ومقصودين، مش تكرار غلط.

**المرحلة 0 — الأساس (بنية البيانات، بدون أي ذكاء بعد):**
- [x] **جدول `product_daily_stats`** — ✅ **تم (17 يوليو 2026)**: views/add_to_cart/purchases/revenue يومي لكل منتج، بـ unique(store_id, product_id, stat_date) + RLS مبنية على owner_id. لسا فاضي — لازم نبني الكود اللي يعبّيه (من GA4 + orders).
- [x] **جدول `store_growth_events`** — ✅ **تم (17 يوليو 2026)**: سجل كل تشخيص/قرار (event_type, category, status: pending/approved/rejected/auto_executed/reverted) + RLS. جاهز يُستخدم من المرحلة 1.
- [x] **جدول `market_benchmarks`** — ✅ **تم (17 يوليو 2026)**: معبّى بـ11 معيار فعلي (Conversion Rate global/fashion/beauty، Meta Ads CTR/CPC/ROAS، TikTok Ads CTR/CPC، Cart Abandonment global/MENA، WhatsApp recovery rate). Migration: `growth_agent_phase0_foundation`.
- [x] كود يعبّي `product_daily_stats` فعلياً — ✅ **تم (17 يوليو 2026)**: `api/growth-agent-sync.js` — يجمع views من GA4 (نفس نمط analytics-data.js) + purchases/revenue من orders (أونلاين) وoffline_sales (بالمحل)، ويعمل upsert يومي. مجدول عبر Vercel Cron الساعة 2 صباحاً UTC (`vercel.json`). قابل للاستدعاء يدوياً: `GET /api/growth-agent-sync?date=YYYY-MM-DD`.
  - **⚠️ اكتشاف مهم أثناء البناء:** بعض عناصر `orders.items[].product_id` قيم قديمة (slug نصي زي `lumia`, `naqaa`) مش UUID فعلي بجدول `products` — على الأغلب بيانات قديمة من قبل توحيد الـ schema. الكود يتجاهلها بأمان (يعدّها بـ `unmatchedOrderItems` بدل ما يفشل)، بس هاد معناه إحصائيات المبيعات لبعض الطلبات القديمة ما رح تنعكس بـ `product_daily_stats`. يستاهل مراجعة لاحقاً لو الرقم طلع كبير.
  - **ملاحظة:** `add_to_cart` مش متتبّع حالياً (يحتاج GA4 event إضافي) — الحقل موجود بالجدول بقيمة 0 مؤقتاً.
- [x] كارت/قسم بالـ Admin يعرض البيانات الخام — ✅ **تم (17 يوليو 2026)**: صفحة `/admin/growth-agent` (رابط جديد بالسايدبار "وكيل النمو") — اختيار متجر، عرض جدول أداء المنتجات آخر 14 يوم (views/purchases/revenue) مع الإجماليات، وجدول معايير السوق الـ11 كاملة. Build اتأكد ينجح (`npm run build` نظيف، صفر أخطاء).

**🎉 المرحلة 0 كاملة بالكامل.** الجداول جاهزة، كود المزامنة اليومية مبني ومجدول عبر Cron، وواجهة التحقق بالـ Admin شغالة. الخطوة الجاية: التأكد إنه أول تشغيل فعلي لـ `growth-agent-sync` نجح (لما يوصلك بيانات حقيقية بالصفحة)، وبعدها نبدأ المرحلة 1 (التشخيص القائم على القواعد).

> ⚠️ **ملاحظة أمان اكتُشفت أثناء الشغل (17 يوليو 2026):** جدول `webhook_logs` بدون RLS مفعّل — أي حد معه anon key يقدر يقرأ/يعدّل عليه. لازم قرار من عثمان قبل التفعيل (تفعيل RLS بدون سياسات بيوقف الـ webhooks الحالية من Togo). لسا مفتوح.

**المرحلة 1 — التشخيص القائم على القواعد (بعد المرحلة 0):**
- [x] **محرك التشخيص** — ✅ **تم (17 يوليو 2026)**: `api/growth-agent-diagnose.js` — يحلل بيانات آخر 14 يوم لكل متجر نشط ويسجل تشخيصات بجدول `store_growth_events`. 4 قواعد مبنية فعلياً:
  1. **منتج راكد** — صفر مبيعات خلال 14 يوم + عمره ≥14 يوم + لسا ظاهر → اقتراح خصم تصريف أو إخفاء
  2. **تحويل ضعيف لمنتج** — مشاهدات كافية (≥30) بس تحويل أقل من نصف متوسط المتجر → المشكلة على الأغلب بصفحة المنتج
  3. **تحويل المتجر ككل ضعيف** — مقارنة مباشرة بمعيار `market_benchmarks` العالمي (لما العينة كافية: ≥50 مشاهدة)
  4. **تخلي عالي عن السلة** — مقارنة `abandoned_carts` مقابل `orders` بمعيار MENA (90%) لما العينة كافية (≥5)
  - كل القواعد فيها **حد أدنى للعينة** قبل ما تحكم (تجنب استنتاج من رقم صغير جداً)
  - **منع التكرار**: نفس التشخيص لنفس المتجر/المنتج ما بيتكرر خلال 7 أيام
  - مجدول Cron يومي الساعة 3 صباحاً UTC (بعد المزامنة بساعة). يدوياً: `GET /api/growth-agent-diagnose?date=YYYY-MM-DD`
- [x] **قرار هندسي:** الصياغة العربية للتشخيص مبنية بقوالب نصية مباشرة بالكود (deterministic)، **مو عبر استدعاء gpt-4.1-nano** كما كان مخطط بالوثيقة الأصلية — توفيراً للتكلفة (تشخيص يومي لعشرات المتاجر عبر AI بيتراكم بسرعة، وهاد كان أصلاً من "التحديات الفعالة" الموثقة بمعرفة المشروع). القوالب واضحة ومباشرة؛ صياغة AI أغنى ممكن تُضاف لاحقاً كتحسين اختياري لو استلزم الأمر.
- [x] **عرض التشخيصات بالـ Dashboard** — ✅ جزئياً: مضافة بصفحة `/admin/growth-agent` (كارت "تشخيصات وكيل النمو") للتحقق الإداري. **لسا لازم تُضاف بـ Dashboard صاحب المتجر نفسه** (مو بس الأدمن) — بند مفتوح.
- [x] **عرض التشخيصات بـ Dashboard صاحب المتجر** — ✅ **تم (17 يوليو 2026)**: صفحة جديدة `/dashboard/growth` ("النمو" بالسايدبار)، مع badge بعدد التشخيصات المستحدثة (آخر 3 أيام) — نفس نمط عداد الطلبات الجديدة. متاحة لأصحاب المتاجر والموظفين بصلاحية `analytics`. Build نظيف (2946 module، صفر أخطاء).
- [ ] **إرسال كإشعار WhatsApp — قرار مؤجل بسبب قيد تقني حقيقي:** ما بنيته. السبب: المنصة حالياً بتستخدم روابط `wa.me` يدوية (يفتحها الموظف بنفسه) — ما في WhatsApp Business API رسمي متكامل يسمح بإرسال تلقائي من السيرفر لصاحب المتجر (هاد أصلاً بند مستقبلي منفصل بالـ "أفكار أكبر" بالـ ROADMAP: "واتساب بيزنس API رسمي"). إرسال تلقائي حالياً غير ممكن تقنياً بدون هاد الـ API. البديل المؤقت المبني فعلياً: صفحة `/dashboard/growth` + badge بالسايدبار.

**المرحلة 2 — ربط Meta + TikTok Ads (بعد المرحلة 1):**
- [x] **البنية التحتية الكاملة مبنية (17 يوليو 2026)** — كود جاهز 100%، لكن **غير فعّال لحد ما توفر حسابات مطورين حقيقية** (تفصيل بالأسفل):
  - جداول `ad_accounts` (بدون RLS write من المتصفح — التوكنات حساسة، فقط السيرفر يكتب)، `ad_campaigns_daily`، `oauth_connect_intents` (مؤقتة، تمنع أي حد يربط حساب إعلانات بمتجر مش تبعه)
  - `api/meta-oauth-start.js` + `api/meta-oauth-callback.js` — تدفق OAuth كامل لـ Meta (تبديل كود → توكن قصير → توكن طويل 60 يوم → جلب أول حساب إعلانات)
  - `api/tiktok-oauth-start.js` + `api/tiktok-oauth-callback.js` — نفس الشي لـ TikTok Business API
  - `api/growth-agent-ads-sync.js` — مزامنة يومية لأداء الحملات (spend/impressions/clicks/CTR/CPC) من كل حساب مربوط، Cron الساعة 2:30 صباحاً UTC
  - قاعدة تشخيص خامسة بـ `growth-agent-diagnose.js`: نسبة نقر ضعيفة مقارنة بمعيار السوق (لكل منصة لحالها)
  - قسم "حسابات الإعلانات" بصفحة `/dashboard/settings` — زر ربط لكل منصة، يتحقق من ملكية المتجر عبر الـ session قبل ما يبدأ الـ OAuth
- [ ] **⚠️ محجوب على إجراء منك خارج الكود بالكامل:** ما في شي تقني تاني أقدر أبنيه هون. لازم:
  1. **Meta:** تنشئ [Meta App](https://developers.facebook.com/apps) نوع Business، تضيف منتج "Facebook Login for Business" + "Marketing API"، وتضيف env vars بـ Vercel: `META_APP_ID`, `META_APP_SECRET`, `META_OAUTH_REDIRECT_URI` (= `https://mawq3i.co/api/meta-oauth-callback`). ملاحظة: طلب صلاحية `ads_read` على تطبيق حي (مو Development mode) بيحتاج **App Review من Meta** — بياخد وقت، وثيقة رسمية مطلوبة توضح استخدام البيانات.
  2. **TikTok:** تسجّل بـ [TikTok for Business Developer Portal](https://business-api.tiktok.com/portal)، تنشئ App، وتضيف env vars: `TIKTOK_APP_ID`, `TIKTOK_APP_SECRET`, `TIKTOK_OAUTH_REDIRECT_URI`.
  3. بعد ما توفر الـ env vars، الكود جاهز يشتغل فوراً بدون أي تعديل إضافي.
- [ ] **قيد معروف موثّق:** الربط الحالي (Meta) بياخد **أول حساب إعلانات** بس من حساب المستخدم (MVP)؛ لو تاجر عنده أكتر من حساب إعلانات، بدنا نضيف واجهة اختيار لاحقاً.
- [ ] **قيد أهم:** ما في ربط حقيقي بين مصروف الإعلان والمبيعة الفعلية (attribution) — لأنه `orders` ما فيها tracking لـ UTM parameters أو `fbclid`. يعني حالياً نقدر نقارن أداء الإعلان (CTR/CPC) بمعيار السوق، بس مو "هاد الإعلان جاب X مبيعة بالضبط". لإضافة attribution حقيقي لازم إضافة capture لـ UTM/click-id بصفحة الهبوط وربطها بـ `orders` — بند مفتوح للمرحلة الجاية لو حبينا نعمقها.

**المرحلة 3 — قرارات تلقائية محدودة (بعد المرحلة 2):**
- [x] **طبقة صلاحيات: تلقائي مقابل يحتاج موافقة** — ✅ **تم (17 يوليو 2026)**: نوعين جدد بمحرك التشخيص:
  - **إجراء مقترح (`suggested_action`)**: منتج راكد باستمرار (≥21 يوم بدون بيعة، بعد ما كان بس "تشخيص" من 14-21 يوم) → status='pending'، بينتظر قرار التاجر
  - **إجراء تلقائي (`auto_action`)**: نفس الحالة، بس لمتجر "موثوق" → يُنفّذ فوراً (إخفاء المنتج) ويتوثّق كـ status='auto_executed'
  - **الأكشن الوحيد المفعّل حالياً: إخفاء منتج راكد.** عمداً ما بنيت غيره (تغيير سعر، إيقاف حملة) — البداية بأقل مخاطرة، قابل التوسع لاحقاً.
- [x] **تعريف "الثقة"** — شرطين لازم يتحققوا الاثنين: عمر المتجر ≥60 يوم (`stores.join_date`) + ≥3 إجراءات مقترحة سابقة وافق عليها التاجر يدوياً (`store_growth_events` بحالة approved). **يعني عملياً: ولا متجر رح يوصل "موثوق" قبل ما يعدي شهرين ويوافق يدوياً على الأقل 3 مرات** — بالضبط زي مبدأ "فترة بناء الثقة" بالوثيقة الأصلية.
- [x] **واجهة موافقة بالـ Dashboard** — ✅ زرين "موافق، نفّذ" و"رفض" بصفحة `/dashboard/growth` لكل إجراء مقترح، وشارة "نُفّذ تلقائياً" للإجراءات التلقائية.
- [x] **`api/growth-agent-execute-action.js`** — endpoint التنفيذ الفعلي. يتحقق من ملكية المتجر عبر الـ session قبل أي تنفيذ. قائمة `action_type` مقفلة ومعروفة بالكود (حالياً `hide_product` بس) — **مافي تنفيذ حر لأي أكشن مو معرّف مسبقاً، هاد أمان مقصود.**
- [x] **كل أكشن قابل للتراجع** — الإخفاء مجرد `status='hidden'`، صاحب المتجر يقدر يرجّعه "ظاهر" من صفحة المنتجات بضغطة وحدة بأي وقت.
- [x] **بند "نتيجة بعد أسبوع"** — ✅ اتبنى، انظر تفصيل المرحلة 4 بالأسفل.

- [x] **🆕 التقييم الشهري لمرحلة النمو + خطة مخصصة (17 يوليو 2026 — طلب مباشر من عثمان)**: `api/growth-agent-monthly-plan.js`، Cron أول كل شهر (5 صباحاً UTC يوم 1).
  - يقارن آخر 30 يوم بالـ30 يوم اللي قبلها (إيراد، طلبات، مشاهدات، تخلي عن سلة، أداء إعلانات لو مربوطة، تكرار كل نوع تشخيص)
  - يصنّف المتجر بمرحلة نمو: **الانطلاق / نمو متسارع / نمو مستقر / ركود / تراجع** (قواعد واضحة: عمر متجر <60 يوم أو <10 طلبات = انطلاق دايماً، وبعدها حسب نسبة تغيّر الإيراد)
  - يبني **خطة أولويات مخصصة فعلياً لكل متجر** (مو نص عام): أعلى نوعين تشخيص تكرروا هالشهر + أولويات خاصة بمرحلة النمو نفسها (مثلاً: متجر "انطلاق" يوصيه يكمّل الكتالوج ويجمع تقييمات، متجر "ركود" يوصيه يجرب قناة تسويق جديدة)
  - **شفافية كاملة**: كل رقم بُني عليه القرار محفوظ بعمود `metrics` — التاجر أو إنت تقدروا تراجعوا "ليش قال هيك" أي وقت
  - يُعرض بأعلى صفحة `/dashboard/growth` — أول شي يشوفه التاجر، قبل التشخيصات اليومية
  - **قرار هندسي:** نفس منطق المرحلة 1 — قوالب نصية ديناميكية مبنية على أرقام حقيقية، مو استدعاء AI لكل متجر شهرياً (توفير تكلفة). لو حسّيت النص جاف بعد ما تشوفه عالأرض، نقدر نغنّيه بطبقة AI اختيارية فوق نفس الأرقام.
- [x] **تمهيد قياس النتائج:** `api/growth-agent-capture-results.js` — لكل إجراء اتنفذ (موافق عليه يدوياً أو تلقائي)، بعد أسبوع بيقارن إيراد المتجر قبل/بعد ويخزّنه بـ `result_snapshot`. Cron أسبوعي (الأحد 4 صباحاً UTC).
  - **مهم:** مقياس على مستوى المتجر ككل، مو معزول تماماً عن باقي العوامل (موسم، عروض تانية...) — إشارة أولية موثّقة بالبيانات نفسها، مو "دليل قاطع" بعد.
- [ ] **"التعلم" الحقيقي (اكتشاف أنماط موثوقة عبر عشرات المتاجر) — لسا مبكر فعلياً وموثّق ليش:** عدد المتاجر الفعليين حالياً أقل بكتير من العتبة اللي حكينا عنها بالوثيقة الأصلية (50-200 متجر) لأي نمط يكون موثوق إحصائياً. بناء هاد الجزء الآن معناه استنتاج من عينة صغيرة جداً — أخطر من مفيد. **القرار الصح: ننتظر لحد ما تتراكم بيانات `result_snapshot` كافية عبر متاجر أكتر.** البنية التحتية (الجدول + آلية القياس) جاهزة بالكامل لهيك اللحظة.
